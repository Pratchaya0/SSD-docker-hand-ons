📦
Docker คืออะไร?

ทำความเข้าใจแนวคิดพื้นฐานก่อนเริ่มใช้งาน

🐳 Container คืออะไร?

Container คือ package ที่บรรจุ Application + ทุกสิ่งที่ต้องการให้รันได้ทันที ไม่ว่าจะอยู่บน Linux, Mac, Windows หรือ Cloud

⚡ เร็วกว่า Virtual Machine

VM ต้องรัน OS เต็มๆ ทำให้หนักและช้า Docker ใช้ kernel ของ host จึงเบากว่า เร็วกว่า และใช้ RAM น้อยกว่ามาก

🌍 Build once, run anywhere

Build image ครั้งเดียว รันได้ทุกที่ที่ติดตั้ง Docker แก้ปัญหา "มันรันได้บนเครื่องฉัน" ตลอดกาล

🔒 แยก Environment สมบูรณ์

แต่ละ container มี environment เป็นของตัวเอง ไม่กระทบ host และไม่กระทบ container อื่น

Components หลักของ Docker

Image — Blueprint ของ container (read-only)  •  Container — Instance ที่กำลังรันจาก Image  •  Registry — คลัง Image (เช่น Docker Hub, ghcr.io)  •  Volume — พื้นที่เก็บข้อมูลถาวร  •  Network — ระบบ network ภายใน

📄
โครงสร้าง Dockerfile

ไฟล์ Dockerfile จริงของโปรเจกต์ SSD Python Starter Template

Dockerfile 
# Python API container image.
# Build: docker build -t python-api .
# Run:   docker compose up --build

FROM python:3.12-slim

ENV PYTHONDONTWRITEBYTECODE=1 \
    PYTHONUNBUFFERED=1 \
    UV_LINK_MODE=copy \
    PYTHONPATH=/app

RUN useradd --create-home appuser
WORKDIR /app

# ติดตั้ง uv จาก distroless image (fast Python package manager)
COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/

# Copy ไฟล์ metadata + source code ก่อน sync
COPY pyproject.toml README.md ./
COPY app ./app
COPY alembic.ini ./alembic.ini
COPY alembic ./alembic
COPY config ./config
COPY scripts ./scripts

# ติดตั้ง production dependencies เท่านั้น
RUN uv sync --no-dev \
    && sed -i 's/\r$//' scripts/docker-entrypoint.sh \
    && chmod +x scripts/docker-entrypoint.sh \
    && chown -R appuser:appuser /app

USER appuser
EXPOSE 8000

ENTRYPOINT ["/app/scripts/docker-entrypoint.sh"]

FROM python:3.12-slim

ใช้ base image ที่เล็กที่สุดเท่าที่จำเป็น (-slim ลดขนาดได้มาก) ไม่มี build tools ที่ไม่จำเป็น

RUN useradd appuser

สร้าง non-root user เพื่อความปลอดภัย Container ที่รันด้วย root มีความเสี่ยงสูงหาก exploit ได้

COPY --from=uv

ใช้ multi-stage COPY เพื่อดึง uv binary จาก official image โดยไม่ต้องติดตั้ง pip หรือ curl

uv sync --no-dev

ติดตั้งเฉพาะ production dependencies ไม่รวม dev tools ทำให้ image เล็กและปลอดภัยขึ้น

✅
Best Practices ใน Dockerfile

หลักปฏิบัติที่ดีที่โปรเจกต์นี้ทำตาม

ใช้ slim image

เลือก python:3.12-slim แทน python:3.12 ลดขนาด image ได้ ~600MB โหลดเร็วขึ้น deploy เร็วขึ้น

สร้าง non-root user (appuser)

รัน container ด้วย user ที่ไม่ใช่ root ลดความเสี่ยงกรณี container ถูก exploit

Copy dependency files ก่อน source code

Docker cache layer ตามลำดับ COPY — ถ้า pyproject.toml ไม่เปลี่ยน layer ของ uv sync จะถูก cache ทำให้ build เร็วมากขึ้น

ติดตั้งเฉพาะ production dependencies

uv sync --no-dev ไม่ติดตั้ง pytest, mypy, black หรือ dev tools อื่นๆ image เล็กลงและไม่มี surface attack เพิ่ม

ใช้ ENTRYPOINT สำหรับ startup script

ENTRYPOINT ให้ความยืดหยุ่นกว่า CMD สำหรับ script ที่ต้องทำ migration ก่อน start server

ใช้ .dockerignore

ไม่รวม .venv, .git, tests/, .cursor/ ใน build context ทำให้ docker build เร็วขึ้นมาก

⌨️
Docker Commands พื้นฐาน

คำสั่งที่ใช้บ่อยสำหรับจัดการ Image และ Container

Terminal 
# ── Build & Run ──────────────────────────────
docker build -t python-api .               # build image
docker run   -p 8000:8000 python-api       # รัน container
docker run   -d -p 8000:8000 python-api    # รันใน background

# ── Inspect ──────────────────────────────────
docker ps                                   # ดู running containers
docker ps -a                               # ดูทุก container (รวม stopped)
docker logs <container_id>                 # ดู logs
docker logs -f <container_id>              # ติดตาม logs แบบ real-time
docker inspect <container_id>              # ดูรายละเอียดทั้งหมด

# ── Shell ─────────────────────────────────────
docker exec -it <container_id> sh          # เข้า shell ใน container
docker exec -it <container_id> bash        # เข้า bash (ถ้ามี)

# ── Stop & Remove ────────────────────────────
docker stop <container_id>                 # หยุด container
docker rm   <container_id>                 # ลบ container
docker rm   -f <container_id>              # หยุดและลบพร้อมกัน

# ── Images ───────────────────────────────────
docker images                              # ดู images ทั้งหมด
docker rmi  python-api                     # ลบ image
docker system prune -a                     # ล้าง image/container ที่ไม่ใช้
🔧
Docker Compose คืออะไร?

เครื่องมือจัดการหลาย containers พร้อมกันใน 1 ไฟล์

📄 1 ไฟล์ ทุก Service

กำหนด api, postgres, nats และทุกอย่างใน compose.yml ไฟล์เดียว version control ได้ง่าย

🔗 จัดการ Dependencies

depends_on บอก Compose ว่าต้องรอ service ไหนให้ healthy ก่อน เช่น api รอ postgres

🌐 Network อัตโนมัติ

Compose สร้าง network ภายในให้อัตโนมัติ ทุก service คุยกันผ่านชื่อ service ได้ทันที

💾 Volume อัตโนมัติ

ประกาศ named volumes ระดับ top-level Compose จัดการให้ทั้งหมด ไม่ต้องสร้าง volume เอง

🚀
คำสั่งหลัก

docker compose up --build  —  Build image ใหม่และรัน stack ทั้งหมด  |  docker compose down  —  หยุดและลบ containers ทั้งหมด

📋
compose.yml ของโปรเจกต์

ไฟล์ compose.yml จริงที่ใช้รัน 3 services: api + postgres + nats

compose.yml 
services:
  api:
    build: .
    ports:
      - "8000:8000"
    environment:
      APP_ENV: docker
      LOG_LEVEL: INFO
      DATABASE_URL: postgresql+asyncpg://app:app@postgres:5432/app
      BROKER_URL: nats://nats:4222
      ENABLE_SCHEDULER: "true"
    depends_on:
      postgres:
        condition: service_healthy    # รอให้ postgres ผ่าน healthcheck
      nats:
        condition: service_started
    restart: unless-stopped

  postgres:
    image: postgres:16-alpine
    ports:
      - "${POSTGRES_HOST_PORT:-5433}:5432"
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app
      POSTGRES_DB: app
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d app"]
      interval: 5s
      timeout: 3s
      retries: 10
    restart: unless-stopped

  nats:
    image: nats:2-alpine
    command: ["-m", "8222"]
    restart: unless-stopped

volumes:
  postgres_data:                            # named volume — docker ดูแลให้
⚙️
Environment Variables & Configuration

ตัวแปร environment ที่ใช้ใน stack และลำดับความสำคัญ

Variableค่าใน compose.ymlคำอธิบาย
DATABASE_URLpostgresql+asyncpg://app:app@postgres:5432/appต่อ PostgreSQL ใน stack
BROKER_URLnats://nats:4222ต่อ NATS broker
APP_ENVdockerระบุ environment
LOG_LEVELINFOระดับ log
ENABLE_SCHEDULERtrueเปิด APScheduler
⚠️
ลำดับความสำคัญ (สูง → ต่ำ)

.env file → process environment → compose.yml → config/settings.yaml → built-in defaults
ห้ามใส่ password, private key หรือ client secret ใน compose.yml — ใช้ .env file หรือ secret store สำหรับ production

🚀
Entrypoint Script — ลำดับการทำงาน

scripts/docker-entrypoint.sh รันทุกครั้งที่ container เริ่มทำงาน

scripts/docker-entrypoint.sh 
#!/usr/bin/env sh
# Container entrypoint: apply DB migrations, then exec the API server.
set -eu

if [ -n "${DATABASE_URL:-}" ]; then
  echo "[entrypoint] Applying Alembic migrations..."
  uv run alembic upgrade head           # ← apply migrations ก่อนเสมอ
else
  echo "[entrypoint] DATABASE_URL not set; skipping migrations."
fi

echo "[entrypoint] Starting API on 0.0.0.0:8000"
exec uv run uvicorn app.main:app \
    --host 0.0.0.0 \
    --port 8000

postgres พร้อมก่อน (depends_on + healthcheck)

Compose ไม่เริ่ม api จนกว่า postgres จะผ่าน pg_isready ทำให้ alembic เชื่อมต่อได้ทันที

uv run alembic upgrade head

Apply database migrations อัตโนมัติทุกครั้งที่ deploy ไม่ต้องรัน migration แยกต่างหาก

exec uvicorn app.main:app

รัน FastAPI server ด้วย exec เพื่อให้ uvicorn เป็น PID 1 รับ signal ได้ถูกต้อง (graceful shutdown)

Docker Volume คืออะไร?

พื้นที่เก็บข้อมูลถาวรที่แยกออกจาก container lifecycle

⚠️
ปัญหาหากไม่ใช้ Volume

ข้อมูลทั้งหมดใน container จะหายไปเมื่อ container ถูกลบหรือ recreate — ไม่ว่าจะเป็น database, uploaded files, หรือ logs

Named Volume

🐳 Docker ดูแลให้

Docker จัดการตำแหน่งจัดเก็บบน host เอง เหมาะสำหรับ database data ที่ต้องการความถาวร

volumes:
  postgres_data:
Bind Mount

📁 Mount จาก Host

Map โฟลเดอร์บน host เข้าไปใน container เหมาะสำหรับ development หรือ log files

volumes:
  - ./logs:/app/logs
tmpfs

⚡ In-Memory

เก็บข้อมูลใน RAM เท่านั้น เร็วมากแต่หายเมื่อ container หยุด เหมาะสำหรับ temp files

tmpfs:
  - /tmp
🗄️
Volume ในโปรเจกต์นี้

การใช้งาน Volume จริงใน SSD Python Starter Template

compose.yml — Volume section 
# Named volume สำหรับ PostgreSQL data
services:
  postgres:
    volumes:
      - postgres_data:/var/lib/postgresql/data

  # Bind mount สำหรับ logs (เพิ่มได้ใน api service)
  api:
    volumes:
      - ./logs:/app/logs   # เก็บ log ข้ามการ restart

# ประกาศ named volume ระดับ top-level
volumes:
  postgres_data:       # Docker จัดการให้อัตโนมัติ

ดู Volumes

docker volume ls
docker volume inspect postgres_data

ลบ Volume (ล้างข้อมูล)

docker compose down -v

Docker Network คืออะไร?

ระบบ network ภายในที่ให้ containers คุยหากัน

🌉 bridge (Default)

Network mode เริ่มต้นสำหรับ Docker Compose ทุก container ในโปรเจกต์เดียวกันอยู่ใน bridge network เดียวกัน คุยกันผ่านชื่อ service ได้

🔗 host

Container ใช้ network ของ host โดยตรง ไม่มี isolation เหมาะสำหรับ performance-sensitive scenarios

🚫 none

ไม่มี network เลย container ถูก isolate สมบูรณ์ เหมาะสำหรับ batch processing ที่ไม่ต้องการ network

Backup — สำรองข้อมูล PostgreSQL

วิธีสำรองและ restore ข้อมูล PostgreSQL ผ่าน docker exec

Backup Commands 
# ── Backup SQL (plain text) ──────────────────────
docker exec postgres \
    pg_dump -U app app \
    > backup_$(date +%Y%m%d).sql

# ── Backup binary format (เร็วกว่า, เล็กกว่า) ─────
docker exec postgres \
    pg_dump -U app -Fc app \
    > backup_$(date +%Y%m%d).dump

# ── Restore จาก SQL ──────────────────────────────
docker exec -i postgres \
    psql -U app app \
    < backup_20260515.sql

# ── Restore จาก binary ───────────────────────────
docker exec -i postgres \
    pg_restore -U app -d app \
    < backup_20260515.dump

⏰ Automated Backup ด้วย cron

เพิ่ม cron job บน host server เพื่อ backup อัตโนมัติทุกวัน

# crontab -e
0 2 * * * docker exec postgres \
  pg_dump -U app app \
  > /backups/db_$(date +%Y%m%d).sql

☁️ อัปโหลดไป Cloud Storage

ส่ง backup ไปยัง S3, GCS หรือ Azure Blob Storage อัตโนมัติหลัง backup

# ส่งไป S3
aws s3 cp backup.sql \
  s3://my-bucket/db/
🚨
กฎการ Backup

1. เก็บ backup นอก container เสมอ — container อาจถูกลบได้ตลอดเวลา  |  2. ทดสอบ restore สม่ำเสมอ — backup ที่ restore ไม่ได้คือไม่มี backup  |  3. เก็บ backup ไว้ อย่างน้อย 3 ที่ (3-2-1 rule)

📦
Backup — สำรอง Volumes และ Logs

วิธีสำรอง Named Volume และ log files

Volume Backup Commands 
# ── Backup Named Volume → tar.gz ─────────────────
docker run --rm \
    -v postgres_data:/data \
    -v $(pwd):/backup \
    alpine \
    tar czf /backup/pg_data_$(date +%Y%m%d).tar.gz /data

# ── Restore Volume จาก tar.gz ─────────────────────
docker run --rm \
    -v postgres_data:/data \
    -v $(pwd):/backup \
    alpine \
    tar xzf /backup/pg_data_20260515.tar.gz -C /

# ── Backup Log files (Bind Mount) ────────────────
cp -r ./logs ./logs_backup_$(date +%Y%m%d)

# ── Backup config files ───────────────────────────
tar czf config_backup.tar.gz \
    config/settings.yaml \
    .env \
    compose.yml

Restart Policy — การรีสตาร์ทอัตโนมัติ

กำหนดพฤติกรรมเมื่อ container crash หรือ server reboot

restart: no

ไม่รีสตาร์ทอัตโนมัติในทุกกรณี (default)

เหมาะสำหรับ: dev, one-off tasks

restart: unless-stopped

รีสตาร์ทเสมอ ยกเว้นถ้า stop ด้วยมือ (docker stop)

✅ แนะนำสำหรับ production

restart: always

รีสตาร์ทเสมอแม้แต่ stop ด้วยมือก็ยังกลับมา

⚠️ ระวัง: อาจ restart loop

restart: on-failure

รีสตาร์ทเฉพาะเมื่อ exit code ไม่ใช่ 0 (crash)

เหมาะสำหรับ: batch jobs

🔍
โปรเจกต์นี้ใช้ unless-stopped

ทั้ง api และ postgres ใช้ restart: unless-stopped ร่วมกับ healthcheck ทำให้ Compose รู้ว่า service พร้อมใช้งานจริงก่อนเริ่ม service ที่ depends on

Deploy บน Hosting Server (VPS/Cloud)

git clone https://github.com/your/repo.git /app

cp .env.example .env
nano .env  # ใส่ secrets จริง

รัน Stack ใน background

docker compose up -d --build
docker compose ps          # ตรวจสอบสถานะ
docker compose logs -f     # ดู logs

ตั้งค่า Nginx / Caddy เป็น Reverse Proxy

api.example.com {
    reverse_proxy localhost:8000
}

Firewall Setup

เปิดเฉพาะ port ที่จำเป็น: 22 (SSH), 80 (HTTP), 443 (HTTPS) — ปิด port 8000 และ 5433 ไม่ให้เข้าถึงจาก internet โดยตรง

Auto-Start หลัง Server Reboot

ทำให้ stack รันอัตโนมัติทุกครั้งที่ server เริ่มใหม่

✅ วิธีที่ 1: restart policy (ง่ายที่สุด)

Docker daemon เริ่มอัตโนมัติเมื่อ OS boot และ restart: unless-stopped จะรัน container กลับมาเอง

# ตรวจสอบว่า docker daemon auto-start
sudo systemctl enable docker
sudo systemctl status docker

🔧 วิธีที่ 2: systemd service

สร้าง systemd service สำหรับ docker compose เพื่อควบคุมได้ละเอียดกว่า

# /etc/systemd/system/myapp.service
[Unit]
After=docker.service

[Service]
WorkingDirectory=/app
ExecStart=docker compose up
ExecStop=docker compose down
Restart=always

[Install]
WantedBy=multi-user.target
ทดสอบ Auto-Start 
# Enable และ start service
sudo systemctl enable myapp
sudo systemctl start myapp

# ทดสอบด้วย reboot จริง
sudo reboot

# หลัง reboot — ตรวจสอบ
docker compose ps          # ต้องเห็น services running
docker compose logs --tail=20

Docker Compose Cheatsheet

คำสั่งที่ใช้บ่อยสรุปไว้ในที่เดียว

docker compose up --build
Build image ใหม่และรัน stack
docker compose up -d
รัน stack ใน background
docker compose down
หยุด stack (เก็บ volumes)
docker compose down -v
หยุด stack + ลบ volumes
docker compose logs -f
ดู logs แบบ real-time
docker compose logs -f api
ดู logs เฉพาะ api service
docker compose ps
ดูสถานะ services ทั้งหมด
docker compose restart api
รีสตาร์ทเฉพาะ api service
docker compose exec api sh
เข้า shell ใน api container
docker compose pull
ดึง image ใหม่จาก registry
docker volume ls
ดู volumes ทั้งหมด
docker volume inspect postgres_data
ดูรายละเอียด volume
docker network ls
ดู networks ทั้งหมด
docker system prune -a
ล้าง images/containers ที่ไม่ใช้

สรุปและ Next Steps

สิ่งที่ได้เรียนรู้และแนวทางต่อไป

🐳 Docker

  • แยก environment ออกจาก host ได้สมบูรณ์
  • Build once, run anywhere
  • เร็วกว่าและเบากว่า VM
  • ใช้ slim image + non-root user

🔧 Docker Compose

  • จัดการ multi-service stack ใน 1 ไฟล์
  • depends_on + healthcheck ป้องกัน race condition
  • Network และ Volume อัตโนมัติ
  • Entrypoint script ทำ migration ก่อน start

💾 Volume & Network

  • Named Volume เก็บข้อมูลถาวร
  • Services คุยกันผ่านชื่อ service
  • Port mapping เปิดให้ host เข้าถึง
  • Backup + restore สม่ำเสมอ

☁️ Production

  • restart: unless-stopped + systemd
  • Nginx/Caddy เป็น reverse proxy
  • Auto-backup ด้วย cron job
  • ทดสอบ restore backup สม่ำเสมอ
